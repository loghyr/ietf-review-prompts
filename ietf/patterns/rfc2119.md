# RFC 2119 / BCP 14 Normative Language Patterns

## Boilerplate Check

The Requirements Language section MUST contain the exact BCP 14 boilerplate.
For drafts using the kramdown-rfc `{::boilerplate bcp14-tagged}` macro, the
macro itself is sufficient — do not flag it as missing text.

For drafts with hand-written boilerplate, verify it cites BOTH RFC 2119 and
RFC 8174 (the update).  Citing only RFC 2119 is a common mistake.

Correct form (abbreviated):
  The key words "MUST", "MUST NOT", "REQUIRED", "SHALL", "SHALL NOT",
  "SHOULD", "SHOULD NOT", "RECOMMENDED", "NOT RECOMMENDED", "MAY", and
  "OPTIONAL" in this document are to be interpreted as described in
  BCP 14 [RFC2119] [RFC8174] when, and only when, they appear in all
  capitals, as shown here.

## Key Word Usage

### MUST / MUST NOT / REQUIRED / SHALL / SHALL NOT
Used for absolute requirements.  Check:
- Is the subject of the requirement clear? (client, server, implementation)
- Is the requirement actually absolute, or is it context-dependent?
  If context-dependent, SHOULD or MAY is more appropriate.
- "REQUIRED" is equivalent to "MUST" — prefer "MUST" for actions,
  "REQUIRED" only for features/properties.

### SHOULD / SHOULD NOT / RECOMMENDED / NOT RECOMMENDED
Used for strong recommendations with known valid exceptions.  Check:
- Is there a stated or implied valid reason to deviate?
  If no valid exception exists, it should be MUST.
- "RECOMMENDED" is equivalent to "SHOULD" — prefer "SHOULD".

### MAY / OPTIONAL
Used for truly optional behavior.  Check:
- If the behavior is described as optional but implementations that
  omit it would be non-interoperable, it should be at least SHOULD.

## Common Mistakes

MISTAKE-1: Lowercase key words in normative prose.
  "the client must send" — "must" should be "MUST".
  "implementations should" — "should" should be "SHOULD".
  Flag any lowercase must/should/may/shall/required/recommended in
  normative prose paragraphs (not in code blocks, examples, or titles).

MISTAKE-2: "will" used as a normative term.
  "the server will return NFS4ERR_INVAL" — "will" is not a BCP 14
  key word.  Use "MUST return" or "returns" (descriptive present tense).

MISTAKE-3: "need to" or "has to" used instead of MUST.
  "the client needs to verify" — replace with "the client MUST verify".

MISTAKE-4: Mixing SHOULD and MUST for the same requirement.
  If a requirement is stated as MUST in one section and SHOULD in another,
  flag the inconsistency.

MISTAKE-5: Normative language in the Abstract or Introduction
  background paragraphs.
  The abstract and general background text are descriptive.  Normative
  requirements in those sections are misplaced; they belong in the
  protocol specification sections.

MISTAKE-6: "Note that" followed by a normative requirement.
  "Note that implementations MUST..." — normative requirements should
  stand alone, not be introduced with "Note that".
